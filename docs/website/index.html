The Lambda Development Problem

⏱

Slow Feedback Loops

Every code change requires redeployment. Wait 30 seconds to 2 minutes just to test a single line change.

🔨

No Real Debugging

CloudWatch logs are the only debugging tool. No breakpoints, no stepping through code, no local IDE integration.

🎯

Environment Mismatch

Local mocks don't replicate real IAM permissions, VPC configurations, or AWS service integrations.

The Live Lambda Solution

Live Lambda proxies real AWS Lambda invocations to your local machine, letting you execute and debug code with your IDE while using real cloud resources.

How It Works

1

Deploy the Infrastructure

The CLI deploys an AppSync Events API for WebSocket communication and a Lambda Layer containing the Go extension. Your existing Lambdas get the layer attached via a CDK aspect. 

pnpm run dev deploy --profile my-profile
2

Start the Local Server

The local development server connects to AppSync via WebSocket and subscribes to the invocation channel. It waits for events from your Lambda functions. 

pnpm run dev server --profile my-profile
3

Invoke Your Lambda

When your Lambda is invoked (via API Gateway, EventBridge, direct invocation, etc.), the Go extension intercepts the Runtime API call and forwards the event to your machine.

4

Execute & Debug Locally

Your local handler executes with assumed Lambda role credentials. Set breakpoints, inspect state, and iterate instantly. The response flows back through AppSync to Lambda.

vs SST Live

SST Live is tightly coupled to the SST framework. Live Lambda works with any AWS CDK project without requiring migration to a different framework. Both use AppSync Events for transport.

vs Lambda Live Debugger

Lambda Live Debugger uses AWS IoT for message transport, which requires additional setup and IoT permissions. Live Lambda uses AppSync Events with IAM auth—simpler and stays within your existing IAM policies.

vs SAM Local

SAM Local runs Lambda in Docker containers locally, simulating the environment. It doesn't use real cloud invocations, so you miss IAM permission issues and can't test real integrations.

Getting Started

Installation

# Clone the repository
git clone https://github.com/boundlessdigital/live-lambda
cd live-lambda

# Install dependencies
pnpm install

# Build the project (compiles TypeScript + Go extension)
pnpm build

Deploy Infrastructure

# Deploy AppSync API and Lambda Layer
pnpm run dev deploy --profile your-aws-profile

Integrate with Your CDK App

import { LiveLambda } from 'live-lambda'

const app = new cdk.App()

// Apply to all NodejsFunction constructs
LiveLambda.install(app, {
  env: { region: 'us-west-2' }
})

Start Developing

# Start the local development server
pnpm run dev server --profile your-aws-profile

# In another terminal, invoke your Lambda
curl https://your-api-gateway-url/endpoint

Under the Hood

Go Lambda Extension

The extension intercepts calls to the Lambda Runtime API using the AWS_LAMBDA_EXEC_WRAPPER mechanism. It acts as a proxy, forwarding invocation events to AppSync and waiting for responses.

  • Compiled for both amd64 and arm64
  • Minimal cold start overhead (~5-10ms)
  • Conditional build (only recompiles when source changes)

AppSync Events API

AWS AppSync Events provides managed WebSocket infrastructure with IAM authentication. The extension publishes to /live-lambda/requests and subscribes to /live-lambda/response/{request_id}.

  • Sub-millisecond message delivery
  • ~$1 per million messages
  • All traffic within your AWS account

Local Development Server

A TypeScript server connects to AppSync, receives invocation events, and executes your local handler code. It assumes the Lambda's IAM role for authentic permissions.

  • TypeScript source map resolution
  • Hot reload (file watching)
  • Full Lambda context simulation
